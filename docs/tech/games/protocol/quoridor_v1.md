# 路墙棋交互协议 v1

## 基本规则

棋盘由 $9 \times 9$ 个棋格组成，坐标从 0 开始。每方有一枚棋子和十面墙，席位 0 先手，双方轮流行动。

每回合必须且只能选择一种行动：

- 移动自己的棋子一步；
- 消耗一面墙，在棋格之间放置一面墙。

席位 0 的棋子从 `(4,0)` 出发，率先抵达任意 `Y=8` 的棋格即获胜；席位 1 的棋子从 `(4,8)` 出发，率先抵达任意 `Y=0` 的棋格即获胜。棋子不需要到达对面起点 `(4,8)` 或 `(4,0)`，抵达目标边上的任意棋格即可。

路墙棋中没有吃子。双方棋子不能占据同一个棋格，已经放下的墙也不能移动或收回。

## 棋格坐标

棋格坐标写作 `(X,Y)`：

- 左上角是 `(0,0)`，右下角是 `(8,8)`；
- `X` 向右增加，合法范围为 `0 <= X <= 8`；
- `Y` 向下增加，合法范围为 `0 <= Y <= 8`。

因此席位 0 总体向棋盘下方前进，席位 1 总体向棋盘上方前进。不过棋子并不要求始终朝目标边移动，只要符合移动规则，也可以横向或向后移动。

## 墙的坐标与方向

墙位于棋格之间，长度恰好覆盖两个棋格的边。墙使用四个相邻棋格之间的内部交叉点作为中心坐标 `(X,Y)`，所以墙坐标的合法范围是：

```text
1 <= X <= 8
1 <= Y <= 8
```

放墙时还要提供方向 `O`：

- `O=0` 表示竖墙；
- `O=1` 表示横墙。

以中心坐标 `(4,2)` 为例：

- 竖墙 `(4,2,0)` 阻断 `(3,1)` 与 `(4,1)` 之间的移动，同时阻断 `(3,2)` 与 `(4,2)` 之间的移动；
- 横墙 `(4,2,1)` 阻断 `(3,1)` 与 `(3,2)` 之间的移动，同时阻断 `(4,1)` 与 `(4,2)` 之间的移动。

墙坐标描述的是棋格之间的交叉点，不是某个棋格，也不是墙的左端或上端。

## 棋子移动规则

### 普通移动

通常情况下，棋子每次可以向上、下、左、右移动一个棋格，但必须同时满足：

- 目标棋格仍在棋盘内；
- 当前棋格与目标棋格之间没有墙；
- 目标棋格没有被对手棋子占据。

棋子不能斜着普通移动，也不能穿墙。

### 跳过对手

如果对手棋子位于己方上下左右紧邻的棋格，并且两枚棋子之间没有墙，则优先检查能否沿同一方向直线跳过对手。

例如己方位于 `(4,4)`，对手位于 `(4,5)`：如果 `(4,6)` 在棋盘内，并且 `(4,5)` 与 `(4,6)` 之间没有墙，则己方可以直接移动到 `(4,6)`。

### 斜跳

只有在无法直线跳到对手身后的棋格时，才能斜跳到对手的侧面。无法直跳仅包括以下两种情况：

- 对手身后的棋格已经超出棋盘；
- 对手与其身后的棋格之间有墙。

仍以己方 `(4,4)`、对手 `(4,5)` 为例：如果 `(4,6)` 越界或被墙隔开，己方可以尝试移动到 `(3,5)` 或 `(5,5)`。每个斜跳目的地都必须在棋盘内，并且对手所在格与该目的地之间不能有墙。

如果对手身后的棋格可以直跳，就不能改为斜跳。双方棋子之间若本身隔着墙，则不能跳过对手，也不能以对手为支点斜跳。

## 放墙规则

放置一面墙会消耗当前行动方的一面墙。墙用完后仍可移动棋子，但不能继续放墙。

一面新墙必须同时满足以下条件：

- 中心坐标和方向合法；
- 不与已有墙交叉；
- 不与同方向的已有墙发生任何一格长度的重叠；
- 放置后，双方棋子都仍至少存在一条通往各自目标边的路径。

例如：

- 已有横墙 `(4,2,1)` 时，不能再放横墙 `(3,2,1)`、`(4,2,1)` 或 `(5,2,1)`，因为它们至少重叠一段；
- 已有横墙 `(4,2,1)` 时，不能再放竖墙 `(4,2,0)`，因为两面墙会在同一中心点交叉；
- 两面墙只在端点相接而没有重叠是合法的，例如横墙 `(4,2,1)` 与 `(6,2,1)` 可以同时存在。

判断是否仍有通路时，只考虑墙的阻挡，不把另一枚棋子当作永久障碍。允许墙迫使对手绕路，但不允许彻底封死任意一方到达目标边的所有路线。

## 程序通信协议

协议使用 UTF-8 文本和十进制非负整数。字段之间由一个或多个 ASCII 空白字符分隔，每一步占一行。程序在一局中只启动一次，并在整个对局期间保持运行。

### 初始化

程序启动后首先从标准输入读取两个整数：

```text
1 S
```

- `1` 是协议版本，目前固定为 1；
- `S` 是己方席位，只能为 `0` 或 `1`。

棋盘大小、初始位置和墙数由 v1 规则固定，因此初始化中不再单独传递这些数据。

席位 0 是先手，读取初始化信息后立即输出第一步。席位 1 是后手，读取初始化信息后还会先收到席位 0 的第一步，然后才输出自己的第一步。

### 输出一步

移动棋子时输出三个整数：

```text
0 X Y
```

其中 `0` 表示移动棋子，`(X,Y)` 是移动后的目标棋格。

放墙时输出四个整数：

```text
1 X Y O
```

其中 `1` 表示放墙，`(X,Y)` 是墙的中心坐标，`O=0` 表示竖墙，`O=1` 表示横墙。

程序输出一步后必须写入换行并立即刷新标准输出。例如 C++ 可以使用 `std::endl`，或在输出 `\n` 后调用 `std::flush`。

### 接收对手的一步

如果对局尚未结束，评测器会把对手刚完成的一步按原格式写入当前程序的标准输入：

```text
0 X Y
```

或：

```text
1 X Y O
```

程序应据此更新本地棋盘、棋子位置、墙和剩余墙数，然后计算自己的下一步。评测器不会在每回合重复发送完整局面。

输入流结束表示对局已经结束。评测器不发送额外的结束标记，也不会在获胜动作之后要求另一方继续行动。

评测端只接受对应格式所需的全部字段。额外 token、负数、非整数和调试文本都会导致格式错误。调试信息只能写入标准错误，但标准错误仍受沙箱输出限制。

## 交互样例

以下样例展示席位 0 程序的一段通信。以 `>>` 开头的内容是程序从标准输入收到的数据，以 `<<` 开头的内容是程序写到标准输出的数据。

```text
>> 1 0
<< 0 4 1
>> 0 4 7
<< 1 4 2 1
>> 0 3 7
<< 0 3 1
>> 1 5 7 0
<< 0 3 2
```

这段通信依次表示：

- 席位 0 从 `(4,0)` 移动到 `(4,1)`；
- 席位 1 从 `(4,8)` 移动到 `(4,7)`；
- 席位 0 以 `(4,2)` 为中心放置横墙；
- 席位 1 从 `(4,7)` 移动到 `(3,7)`；
- 席位 0 从 `(4,1)` 移动到 `(3,1)`；
- 席位 1 以 `(5,7)` 为中心放置竖墙；
- 席位 0 从 `(3,1)` 移动到 `(3,2)`。

## 终局与判罚

任一棋子首次抵达自己的目标边时立即获胜，对局结束，双方输入流随后关闭。

若双方均已各自完成 100 步，即整局共完成 200 步后仍无人抵达目标边，对局以 `move_limit` 结束。此时用户程序判负，即 `won = false`；只要程序此前始终遵守协议，verdict 仍可为 `ACCEPTED`。当前版本没有其他和棋终局。

以下情况均属于非法操作：

- 输出格式错误、字段数量错误或包含额外内容；
- 棋子目标坐标或墙中心坐标越界；
- 棋子移动不符合普通移动、直跳或斜跳规则；
- 棋子穿墙或移动到对手所在格；
- 墙已经用完却继续放墙；
- 新墙与已有墙重叠或交叉；
- 新墙封死任意一方通往目标边的全部路径；
- 在不属于己方的回合输出行动。

用户程序的非法操作记为 `INVALID_MOVE`。平台内置程序发生非法操作、协议错误、编译失败或运行失败属于平台配置错误，Evaluation 记为 `INTERNAL_ERROR`。

`ACCEPTED` 只表示用户程序遵守协议并正常完成对局，不表示用户一定获胜。胜负结果由 `won` 单独记录。

## 资源限制

双方是两个独立的长驻 C++20 进程。每一步只解冻当前行动方，并分别限制单步 CPU、单方整局累计 CPU、单步墙上时间、内存、输出和进程数。等待对手行动期间，程序处于冻结状态，不消耗活动 CPU 时间预算。

具体的单步 CPU、整局累计 CPU 和内存限制来自游戏目录。单步墙上时间上限为 1 秒，只用于防止程序永久阻塞，不能替代 CPU 限制。

整局墙上时间上限为 300 秒，进程 `clockLimit` 与 Worker 的对局时限保持一致。超过限制记为 `TIME_LIMIT_EXCEEDED`。对局结束后，评测器会终止双方进程并清理临时编译产物。
